# 기능 요청을 받으면 왜 코드부터 치면 안 될까

구현 요청이나 수정 요청을 받으면 일단 코드부터 보는 분이 많을 것 같습니다. 저도 그랬습니다.

판도라티비에 다닐 때 "글쓰기에 임시저장 기능을 추가해주세요."라는 요청을 받은 적이 있습니다. 저는 당시에 별다른 고민 없이 바로 코드 작업에 들어갔습니다.

흐름은 대략 아래와 같았습니다.

```
[요청]: 기존 글쓰기 기능에 임시저장 기능을 추가
[선판단]: 기존 글쓰기 컴포넌트를 재사용하면서 flag 형태의 state를 두고 분기 처리하면 되겠다.
[설계]: 리로딩 시에도 해당 상태가 유지되어야 하니 URL에 draft 상태를 두자.
[구현]: "/write", "/edit" 관련 컴포넌트에 분기 처리 로직 추가 ...
```

기능은 동작했습니다. 하지만 하나의 컴포넌트 안에 URL 상태 기반 분기가 겹겹이 쌓이면서, 작은 수정 하나에도 전체 흐름을 다시 읽어야 하는 코드가 됐습니다. 이후 "임시저장 목록에서 이어쓰기" 같은 추가 요청이 들어왔을 때, 분기를 하나 추가하면 기존 분기가 깨지는 상태였습니다.

지금 생각해보면 코드 퀄리티만의 문제가 아니었습니다. 넓게 들어온 요청을 작업 가능한 단위로 전환하지 않고 그대로 코드에 녹이려다 보니, 정해지지 않은 부분을 순간순간 임의로 판단하면서 짠 코드가 쌓인 것이었습니다.

이 글은 그 이후로 고민하고 학습한, 프론트엔드 개발자가 넓은 요청을 받았을 때 어떻게 작업 단위로 전환하나에 대한 내용을 다뤄보려 합니다.

## "빠르게 해주세요", "예쁘게 해주세요"는 명세가 아니다

기능 요청은 대부분 이렇게 옵니다.

> "결제가 실패해도 사용자가 큰 불편함 없게 해주세요."  
> "장바구니의 동작을 조금 더 빠르게 해주세요."  
> "사용자가 실수 했을 때 치명적이지 않도록 방어해주세요."

자연스러운 말이고, 대부분의 첫 요구사항은 이런 식으로 전달되는 경우가 많습니다. 하지만 이걸 그대로 받아서 코드를 시작하면, 나중에 "제가 원한 건 이게 아닌데요"가 터집니다.

"빠르게", "예쁘게"를 요청받았을 때 분명 주관적으로 떠오르는 감각이나 코드 작성 방향이 있을 수는 있습니다.

하지만 그 감각만 믿고 바로 코드에 들어가면 맨 처음 예시처럼 "정해지지 않은 모호한 구간"에서 무너집니다.

예를 들어 "홈 화면을 예쁘게"라는 요청에 "그라데이션이 촌스러우니까 단색에 가깝게 줄이면 되겠다"라고 판단할 수는 있습니다. 하지만 어느 정도로 줄일지, 그라데이션에 맞춰 잡아둔 텍스트 색상은 어떻게 할지 같은 후속 결정이 계속 생깁니다. 그때마다 추측으로 해결해야 합니다.

"빠르게"도 마찬가지입니다. "장바구니 로딩이 느리니까 API 응답을 캐싱하면 되겠다"라고 판단할 수는 있습니다. 하지만 어느 정도 빨라야 충분한지, 버튼 클릭 반응인지 페이지 전체 로딩인지, 측정 기준은 평균인지 p95인지가 정해져 있지 않으면 같은 문제가 반복됩니다.

그래서 SPEC(구현 명세)이 필요합니다. SPEC은 모호한 표현을 검증 가능한 조건으로 바꾸는 작업입니다.

```
"빠르게"
-> 버튼을 누른 뒤 300ms 안에 "처리 중" 피드백이 보인다 (p95 기준)
-> 결제 완료 화면은 5초 안에 뜬다 (p95 기준)
```

"빠르게", "예쁘게"같은 형용사를 지우는 게 아닙니다. 그 형용사가 가리키는 리스크를 쪼개서, 나중에 "됐는지 안 됐는지" 확인할 수 있는 조건으로 바꾸는 것입니다.

### SPEC의 8가지 필드

저는 기능 요청이 들어오면 이 8가지를 채워봅니다. 다 채우지 못할 때도 있지만, 채운 만큼 "나중에 터지는 문제"가 줄었습니다.

여기서 특히 두 가지가 중요합니다.

Non-goal을 명시하는 것. "결제 flow 속도는 이번에 안 건드린다"를 적어두지 않으면, 나중에 "그것도 당연히 포함 아니었어요?"라는 분쟁이 생깁니다. 빠진 것을 명시하는 것도 명세의 일입니다.

다음 글에서 다루겠지만, AI 코딩 도구가 의도하지 않은 작업까지 "하는 김에" 처리하는 것을 차단하는 원리와도 비슷합니다.

TBD에 이름과 기한을 다는 것. 결정되지 않은 항목을 "TBD"로만 적어두면, 개발자가 혼자 추측해서 채워버립니다. "아마 이렇겠지"로 만든 코드는 나중에 "이거 누가 이렇게 하라고 했어요?"가 됩니다.

반대로 TBD를 남발하는 것도 좋지 않습니다. 모든 항목을 TBD로 미뤄두면 결정이 계속 밀리고, 결국 개발자가 추측해서 채우게 됩니다.

## "완료"의 기준은 화면에 보이는 것만이 아니다

SPEC의 Acceptance 필드를 막상 채우려고 하면, "버튼을 누르면 모달이 뜬다", "목록에 새 항목이 추가된다"처럼 화면에서 보이는 것만 쓰게 됩니다. 하지만 완료 기준은 7가지 카테고리로 나뉩니다.

저 같은 프론트엔드 개발자 입장에서 보면 위 7가지 중 Server authority나 Data consistency는 "백엔드가 처리할 부분 아닌가?"라는 생각이 들 수 있습니다. 하지만 프론트엔드 코드에서도 이 7가지가 전부 영향을 미칩니다.

Server authority는 프론트엔드가 버튼을 disabled로 만드는 것으로 끝나지 않습니다.

- 버튼 disabled - DevTools에서 해제 가능
- localStorage 잠금 - 다른 브라우저에서 접근 가능
- 클라이언트 validation - API를 직접 호출하면 우회 가능

권한 검사, 중복 결제 방지, 재고 확인처럼 잘못되면 실제 돈이나 데이터 문제가 생기는 것은 반드시 서버가 최종 판정해야 합니다. 프론트엔드는 사용자에게 안내하고 도와주는 역할이지, 규칙을 강제하는 역할이 아닙니다.

Client recovery는 `alert('문제가 발생했습니다')`를 띄우는 것이 전부가 아닙니다. 사용자가 작성 중이던 데이터는 어떻게 되는지, 재시도에 대한 UI/UX는 어떻게 처리해야 할지, 부분적으로 성공한 경우에는 어떤 화면을 보여줘야 하는지 등이 남습니다.

이 기준들을 챙기지 않으면 "기능은 만들었는데 이후에 자꾸 문의가 오는" 상황이 될 수 있습니다.

### Given/When/Then으로 적으면 테스트로 옮길 수 있다

완료 기준 7가지를 매번 다 채우기 어려울 수 있습니다. 이럴 때는 Given/When/Then만이라도 적어두면 테스트 코드로 옮길 수 있는 최소 기준이 잡힙니다.

나쁜 acceptance

```
장바구니가 빠르게 동작해야 한다.
```

좋은 acceptance

```
Given 장바구니에 상품이 20개 있을 때,
When 사용자가 수량 변경 버튼을 클릭하면,
Then 300ms 이내에 변경된 수량이 화면에 반영되고
     서버 응답 전까지 낙관적 업데이트가 보인다.
And 서버가 실패를 응답하면 이전 수량으로 롤백된다.
```

첫 번째는 느낌이고, 두 번째는 테스트 코드로 바로 옮길 수 있는 조건입니다. 이 차이가 나중에 "이 기능이 제대로 동작하는지 어떻게 확인하지?"에 대한 답을 결정합니다.

## 화면부터 그리면 왜 문제인가

SPEC을 쓰고, 완료 기준도 나눴습니다. 이제 Scope에 잡아둔 수량 변경을 구현할 차례인데, 여기서 접근 순서가 중요합니다.

### 잘못된 접근 - 화면부터

```
Scope: 수량 변경 낙관적 업데이트
개발자: input UI 만듦 -> onChange 연결 -> 완성!

나중에 발견:
  +버튼 연타 -> 서버에 요청이 5번 감 -> 재고 불일치
  재고 0인데 화면에서는 수량을 올릴 수 있음
  네트워크 에러 시 아무 피드백 없음
  새로고침하면 바뀐 수량 사라짐
```

### 올바른 접근 - 데이터 흐름부터

```
Scope: 수량 변경 낙관적 업데이트
개발자:
  1. 서버에 뭘 쓰나? -> PATCH /cart-items/:id
  2. 권한은? -> 본인 장바구니만
  3. 중복 요청은? -> debounce + 서버 idempotency
  4. 재고 초과는? -> 서버가 거부, 클라이언트가 에러 표시
  5. 실패 복구는? -> 이전 수량 롤백 + 재시도 버튼
  6. 상태 분리: 서버 vs 화면 vs 낙관적 업데이트
  7. 화면 상태 목록: 로딩, 성공, 실패, 재고 초과, 권한 없음
  8. 가장 얇은 end-to-end -> 첫 PR
```

차이가 보입니다. 첫 번째는 화면에서 시작하고, 두 번째는 데이터 흐름에서 시작합니다.

### Intake - 숨어 있는 요구사항 캐내기

Scope에 잡아둔 "수량 변경 낙관적 업데이트"라는 한 줄 뒤에는 이런 것들이 숨어 있습니다.

- 이 변경은 서버에 어떤 데이터를 쓰나? (write path)
- 누가 이 작업을 할 수 있나? (permission)
- 두 명이 동시에 같은 항목을 바꾸면? (conflict)
- 요청이 실패하면 사용자는 뭘 보게 되나? (failure path)
- 비동기 처리가 필요한가? (async)

이걸 코드 시작 전에 캐내는 단계를 Intake라고 합니다. Intake를 건너뛰면, UI를 다 만든 뒤에 "이건 서버에서 막아야 해요"가 튀어나옵니다.

### State Ownership - 상태를 섞지 않기

프론트엔드에서 흔한 버그 중 하나가 상태 혼합입니다.

```
서버 상태      -> 장바구니 내용 (API 응답, source of truth)
클라이언트 상태 -> 모달 열림/닫힘, 현재 탭
임시 입력      -> 사용자가 타이핑 중인 수량 (아직 저장 안 됨)
낙관적 업데이트 -> 서버 응답 전에 미리 보여주는 변경된 수량
```

이 네 가지를 하나의 state에 섞으면 이런 일이 생깁니다.

- 새로고침하면 클라이언트 상태가 사라져서 이상한 화면이 된다
- 서버가 요청을 거부했는데 화면에는 변경된 값이 남아 있다
- 임시 입력과 서버 데이터가 충돌해서 "방금 바꿨는데 왜 안 바뀌었지?"가 된다

서버 데이터, 클라이언트만의 상태, 아직 저장하지 않은 임시 입력, 낙관적 업데이트를 분리해서 관리해야 합니다. TanStack Query나 SWR 같은 라이브러리가 서버 상태를 분리해주는 것도 이 원칙에 기반합니다.

## 코드는 어떻게 나누고 내보낼까

SPEC을 쓰고, 완료 기준을 나누고, 데이터 흐름을 먼저 설계했습니다. 이제 코드를 작성하고 PR을 올릴 차례입니다.

여기서 흔한 두 가지 실수가 있습니다.

**너무 넓은 첫 PR.**

```
Scope 전체를 한 PR에 넣음
  낙관적 업데이트 + 목록 캐싱 + 캐시 무효화 + 에러 상태 세분화
  -> 리뷰어는 수천 줄 앞에서 막막
```

**너무 얇은 첫 PR.**

```
수량 변경 UI만 만듦
  숫자 표시 + ±버튼 + disabled 스타일
  -> 서버 연결 없음, 새로고침하면 초기화
```

화면은 예쁘지만 서버에 아무것도 보내지 않으니, 이 상태로 머지하면 "동작하지 않는 기능"이 배포됩니다.

### Thin Vertical Slice - 딱 맞는 첫 조각

첫 번째 PR은 **사용자 목표 하나가 서버까지 갔다 오는 최소 흐름**이어야 합니다.

```
수량 변경 - Thin Slice
  수량 input UI (단순하게)
  PATCH /cart-items/:id 요청
  서버 응답에 따른 성공/실패 화면
  재고 초과 시 에러 메시지
  동시 요청 방지 (pending 상태 표시)
```

UI가 화려하지 않아도 됩니다. 중요한 건 서버까지 왕복하는 것입니다. 서버 왕복이 있어야 네트워크 에러, 서버 거부, 응답 지연 같은 실제 동작의 문제를 첫 PR에서 잡을 수 있습니다.

나머지는 이 slice가 안정된 뒤에 후속 PR로 추가합니다.

- 목록 캐싱 + 캐시 무효화 -> 별도 slice
- 빈 장바구니 상태, 애니메이션 -> UI 보강 PR
- 에러 상태 세분화 -> 별도 PR

### Release Slicing - 위험 단위로 쪼개기

첫 slice가 안정되고 기능이 완성되면 배포할 차례입니다. 여기서도 한 번에 다 내보내면 위험합니다.

```
한 PR에 전부 넣음
  낙관적 업데이트 로직 + API 응답 변경 + 캐시 전략 변경 + 기존 동기 코드 삭제
  -> 전체 사용자에게 즉시 노출

문제 발생 시
  뭘 되돌려야 할지 모른다
  캐시는 이미 바뀌었는데 코드를 되돌리면 정합성이 깨진다
```

"PR 크기"가 아니라 "위험 격리"를 기준으로 쪼갭니다.

```
PR 1: contract expand
      PATCH /cart-items 응답에 optimistic_version 필드 추가 (기존 동작 영향 없음)

PR 2: hidden implementation
      낙관적 업데이트 로직 추가, feature flag off로 기존 방식 유지

PR 3: guarded exposure
      내부 팀에만 flag on, 에러율/응답 속도 확인

PR 4: ramp
      5% -> 25% -> 100% 단계적 확대

PR 5: cleanup
      flag, 기존 동기 업데이트 코드 제거
```

각 단계가 격리되어 있으니, 3단계에서 문제가 생겨도 flag만 끄면 됩니다. 1단계와 2단계는 영향을 받지 않습니다.

### Expand -> Migrate -> Contract

API 응답 구조를 바꿀 때는 "낡은 것 먼저 삭제"를 하면 안 됩니다.

```
나쁜 순서:
  cart-items API 응답을 v2 형식으로 교체 -> 기존 코드가 v1 필드를 찾다 에러

좋은 순서:
  1. expand   - v2 필드를 optional로 추가 (v1 필드 유지)
  2. migrate  - 프론트엔드가 v2 필드 참조로 전환
  3. contract - v1 필드 제거
```

이 순서를 지키면 중간에 배포가 섞여도 안전합니다. 어느 시점이든 옛 코드와 새 코드가 공존할 수 있기 때문입니다.

### Feature flag - 유용하지만 만능은 아니다

Feature flag는 코드 배포 없이 기능을 켜고 끌 수 있는 스위치입니다. 하지만 flag로 보장할 수 없는 것들이 있습니다.

- 서버 권한 검사 - flag가 UI를 숨겨도 API는 여전히 호출 가능
- DB constraint - flag와 무관하게 데이터는 이미 기록됨
- 이미 발생한 데이터 - flag를 꺼도 이미 쓰인 데이터는 되돌아오지 않음

"flag가 있으니 괜찮겠지"가 아니라, flag가 실제로 멈출 수 있는 범위를 아는 것이 중요합니다.

사실 위에 보기 좋게 이론적인 얘기를 써놓았지만, 매번 이 작업을 직접 하긴 어렵습니다.

다만 이제는 AI가 적극적으로 도입된 환경이 많다 보니 AI와 함께 이렇게 작업을 하면 흔들리지 않는 작업이 가능해질 수 있었습니다.
