## Q2. 위처럼 접근하려면 우선 "뭐가 Goal인가?" / "이건 Goal인가? 아니면 Goal을 위한 과정인가? 하지만 그것도 결국 작은범위의 Goal?" << 이 판단을 기계적으로 할수...

질문 원문:

위처럼 접근하려면 우선 "뭐가 Goal인가?" / "이건 Goal인가? 아니면 Goal을 위한 과정인가? 하지만 그것도 결국 작은범위의 Goal?" << 이 판단을 기계적으로 할수있나요? 아니면 이건 "감"의 영역인가요?

**비유**

- `Goal`은 “도착 상태”, `과정`은 “이동 방법”, `Task`는 “오늘 한 걸음”입니다.
- 작은 범위도 `도착 상태 + 성공판정 숫자 + 기한`이 있으면 **Sub-goal**로 봅니다.

**코드/텍스트 설명**

- 결론: 완전한 감의 영역은 아니고, **기계적 1차 분류(약 80%) + 맥락 판단(약 20%)**이 맞습니다.
- 1차 분류 규칙(Yes/No):

1. 완료 시 사용자/비즈니스 상태가 바뀌는가?
2. 성공을 `지연지표(lagging)`로 측정 가능한가?
3. 상위 목표가 없어도 독립 가치가 남는가?
4. 기한이 있는가?

- 판정:

1. 3~4개 Yes: Goal(또는 Sub-goal)
2. 2개 Yes: 경계선(대부분 Sub-goal 후보)
3. 0~1개 Yes: 과정/Task

프로젝트 예시:

- “Lighthouse CI 도입”은 대체로 **과정**: [docs/review/roadmap.md](/Users/kimyounghoon/Downloads/@kyh/joseph0926/docs/review/roadmap.md):279
- “LCP/INP/CLS 개선”은 **Goal**: [docs/review/roadmap.md](/Users/kimyounghoon/Downloads/@kyh/joseph0926/docs/review/roadmap.md):291
- “coverage threshold 설정”은 기본적으로 **과정/가드레일**, 다만 기한+성과 기준이 붙으면 **Sub-goal**: [apps/blog/vitest.config.ts](/Users/kimyounghoon/Downloads/@kyh/joseph0926/apps/blog/vitest.config.ts):28, [docs/review/roadmap.md](/Users/kimyounghoon/Downloads/@kyh/joseph0926/docs/review/roadmap.md):252

추론: “Goal인가 과정인가”는 절대 분류가 아니라, **계층(상위 Goal ↔ 하위 Goal ↔ 과정)**으로 보는 게 실무에서 가장 정확합니다.

**시각화**

```text
항목 X
 ├─ 외부 상태 변화 있음? ─ 아니오 -> 과정/Task
 │
 └─ 예
    ├─ 지연지표로 성공판정 가능? ─ 아니오 -> 과정(지표 설계 필요)
    │
    └─ 예
       ├─ 독립 가치 + 기한 있음 -> Goal/Sub-goal
       └─ 둘 중 하나 없음 -> Sub-goal 후보(정의 보강)
```

## Q3. 지연지표로 성공판정 << 이게 무슨 뜻인가요?

`지연지표(lagging indicator)`는  
“행동을 한 뒤, 시간이 지나 **결과로 나타나는 숫자**”라는 뜻입니다.

예시로 보면:

- 리딩지표(선행): `E2E 테스트 케이스 수`, `코드리뷰 리드타임`
- 지연지표(결과): `배포 후 장애율`, `CI 실패율`, `LCP/INP/CLS`

즉 “성공 판정”을 지연지표로 한다는 건,  
“우리가 잘했는지”를 **최종 결과 숫자**로 판단한다는 뜻입니다.  
리딩지표는 조정용 핸들이고, 지연지표는 최종 스코어라고 보면 됩니다.
